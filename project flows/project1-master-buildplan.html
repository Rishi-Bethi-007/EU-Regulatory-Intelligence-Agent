Week 1
01
Core Agent + Supabase + Streamlit Shell
🏗 Core Wiring
Goal: One working LangGraph agent hitting live web search. Every run logged to Supabase. Basic Streamlit input wired up.
1.1
Build Supabase client singleton in db/client.py
Single get_supabase_client() function reading from .env. Import this everywhere — never instantiate more than once. Also build db/queries.py with typed helper functions for every table operation you'll use throughout the project.
1.2
Build first LangGraph agent — single Researcher node + Brave Search tool
In agents/researcher.py: minimal StateGraph with one node. Give it one tool: Brave Search via direct HTTP (not MCP yet). Confirm it returns a real live web result before adding complexity. Keep this ugly — correctness first, architecture second.
1.3
Log every agent run to Supabase research_runs — from the very first run
Before agent: INSERT INTO research_runs (goal, status, user_id). After agent: UPDATE SET status='completed', total_tokens=N, cost_usd=N. Token count from LLM response metadata. This habit from run #1 means your audit trail is complete for the entire project with zero retrofitting.
1.4
Set up LangSmith tracing — verify it's capturing on first run
Add to .env: LANGCHAIN_TRACING_V2=true, LANGCHAIN_PROJECT=eu-reg-agent, LANGCHAIN_API_KEY=ls__.... Open smith.langchain.com after your first agent run — you should see the trace automatically. If you don't see it, fix it now before adding more agents.
1.5
Build GDPR middleware in compliance/gdpr_middleware.py
Three functions: record_consent(user_id, purpose) that inserts to consent_records, scrub_pii(text) that strips email/phone/EU-ID patterns before ingestion, and request_erasure(user_id) that anonymizes all user data. Wire record_consent to the Streamlit login flow now — even before auth is built, call it on session start.
1.6
Build Streamlit shell — Page 1 (Home) with research input
One st.text_area for goal, one st.button("Run Research"). On click: call agent directly (no FastAPI yet), show st.spinner(), display result with st.write(). Keep it ugly. The point is end-to-end wiring: UI → Agent → Supabase → UI.
✓ Phase 1 Complete When: You type a research goal, click Run, see a real LLM response with live web search results, see a new row in Supabase research_runs with token count and cost, and see the trace in LangSmith. The GDPR consent record is also written.
Week 1–2
02
RAG Pipeline + Multilingual Embeddings
📚 RAG + NLP
🌍 Swedish + German
Goal: EU AI Act PDF ingested, retrievable in English, Swedish, and German. First RAGAS baseline score recorded.
2.1
Build ingestion pipeline in rag/ingest.py with PII scrubbing
PyPDFLoader for PDFs, WebBaseLoader for URLs. Run each doc through scrub_pii() before chunking. Use SemanticChunker(HuggingFaceEmbeddings("intfloat/multilingual-e5-large")) — splits by meaning, not character count. Significant quality difference. Test manually on EU AI Act PDF before moving on.
2.2
Add language detection to ingestion — route to correct embedding model
After loading each document: from langdetect import detect; lang = detect(doc.page_content[:500]). Store as metadata["language"] on each chunk. Add language column to document_chunks table. When the query is in Swedish, prioritise Swedish chunks. When German, prioritise German. This 20-line addition is your multilingual differentiator.
2.3
Write chunks to Supabase pgvector with SupabaseVectorStore
SupabaseVectorStore.from_documents(chunks, HuggingFaceEmbeddings("intfloat/multilingual-e5-large"), client=supabase, table_name='document_chunks', query_name='match_chunks'). Note: update your pgvector column to vector(1024) since multilingual-e5-large is 1024-dim (not 1536 like OpenAI). Verify rows in Supabase after each document.
2.4
Build hybrid retriever in rag/retriever.py (pgvector + BM25)
Subclass BaseRetriever. Run pgvector similarity search AND Postgres full-text search in parallel. Merge and deduplicate by source. Add language filter: if detect(query) == 'sv', boost chunks where language='sv'. This language-aware retrieval is your key multilingual feature — test it by querying in Swedish.
2.5
Test multilingual retrieval on Swedish DIGG PDF — screenshot this
Upload digg_ai_vagledning_sv.pdf via Streamlit. Query: "Vad säger DIGG om AI-reglering?" — verify you get Swedish chunks back. Then query the same topic in English — verify you get both languages. Screenshot both results. This screenshot goes on LinkedIn when you announce the project. It's your most visible Sweden-specific proof point.
2.6
Run first RAGAS baseline evaluation — record the number
Write 10 EU AI Act Q&A pairs manually in evals/qa_pairs_manual.json. Run evaluate(dataset, metrics=[faithfulness, answer_relevancy, context_precision]). Store scores in Supabase eval_scores. This number is your baseline — it doesn't need to be high, it just needs to exist. You'll reference it as the "before" in your before/after comparison.
2.7
Build Streamlit Page 2 — Document Upload with language indicator
st.file_uploader(accept_multiple_files=True, type=['pdf','txt']). On upload: save to Supabase Storage, ingest pipeline runs, show language detected as st.success("Detected: Swedish 🇸🇪"). Add st.progress() for ingestion progress. This page is demo-ready now.
✓ Phase 2 Complete When: You upload a Swedish PDF and query it in both Swedish and English, getting relevant chunks in both cases. RAGAS baseline score is stored in Supabase. LangSmith shows the retrieval steps in every trace.
Week 2–3
03
Multi-Agent Orchestration + EU Compliance Module
🧠 Agents + EU
🇪🇺 Risk Classifier + XAI
Goal: All 5 agents running via LangGraph + A2A. EU AI Act risk classification active. XAI decision traces emitting on every agent decision.
3.1
Build Planner Agent in agents/planner.py
Claude Opus + .with_structured_output(SubTaskPlan) where SubTaskPlan is a Pydantic model: tasks: list[SubTask], each with agent_type, input, parallel: bool. Test decomposition on 5 different EU AI Act research goals before wiring routing. Build incrementally — Planner first, add other agents one at a time.
3.2
Build EU AI Act Risk Classifier in compliance/risk_classifier.py
Load your annex_iii_high_risk_list.json. For each research goal, use Claude with a structured prompt to classify: UNACCEPTABLE / HIGH_RISK / LIMITED_RISK / MINIMAL_RISK per EU AI Act Annex III. Store result as research_runs.risk_level. Call this from the Planner node before decomposing tasks. This is a 50-line module that signals deep EU AI Act understanding.
3.3
Add DecisionTrace Pydantic model to every agent
Create in compliance/xai_traces.py: class DecisionTrace(BaseModel): agent_name: str; reasoning_steps: list[str]; sources_used: list[str]; confidence: float; alternatives_considered: list[str]. Every agent node returns both its output AND a DecisionTrace. Store as JSONB in agent_tasks.decision_trace. This is EU AI Act Article 13 compliance made concrete.
3.4
Build full LangGraph StateGraph in agents/orchestrator.py
Define shared TypedDict state. Add all 5 nodes. Conditional edges from Planner → route to correct agents. Use Send() API for parallel execution when parallel=True. Every node writes its status + decision_trace to Supabase agent_tasks at start and end. Build Planner+Researcher first. Test. Then add Analyst, Critic, Synthesizer one at a time.
3.5
Implement A2A Protocol between Planner and all specialists
Define AgentCard for each specialist (name, capabilities, input/output schema). Planner emits Task messages to each specialist's endpoint. Start with Researcher only, verify A2A messaging works, then extend to all agents. A2A makes every agent-to-agent message a trackable, loggable event.
3.6
Build Analyst, Critic (GPT-4o judge + counterfactual), and Synthesizer agents
Analyst: Claude Sonnet + structured Pydantic output + chart generation. Critic: GPT-4o with confidence 0–1 scoring per claim + counterfactual generation when score <0.7 (triggers retry). Synthesizer: Claude Sonnet + Jinja2 template + EU AI Act transparency notice appended to every report. Each agent writes its DecisionTrace.
3.7
Build immutable audit trail in compliance/audit_log.py
Every agent event calls write_audit_event(run_id, event_type, payload) which: generates SHA-256 hash of (previous_hash + payload), inserts to append-only audit_events table. Add a DB trigger or application-level check that prevents UPDATE/DELETE on audit_events. This tamper-evident design is EU AI Act Article 12 compliance.
✓ Phase 3 Complete When: Submit "EU AI Act obligations for Swedish manufacturing SME deploying quality control AI." All 5 agents fire in correct order, a report is saved to Supabase Storage, every agent task has a risk level and DecisionTrace, audit_events has a complete hash chain, and LangSmith shows the full multi-agent trace.
Week 3
04
Custom MCP Servers + FastAPI Layer
🔧 MCP + API
Goal: 2 custom MCP servers written by you and connected. FastAPI async layer between Streamlit and agents. Resume differentiator unlocked.
4.1
Build Web Scraper MCP Server in tools/scraper_mcp.py
Use mcp SDK. Expose one tool: scrape_url(url: str) → str. Inside: fetch with httpx, parse with BeautifulSoup, strip nav/footer/ads, return clean markdown. Run as subprocess. Register in Researcher agent via MCPToolkit. This is MCP code you wrote — say that explicitly in your README and interviews.
4.2
Build Citation Formatter MCP Server in tools/citation_mcp.py
Expose: format_citation(url, title, date, excerpt, style="APA") → str. Returns formatted APA or MLA citation string. The Synthesizer calls this before adding citations to the report. Simple but shows you understand writing MCP servers, not just consuming them.
4.3
Connect official Brave Search MCP server
Clone official Brave MCP repo. Set BRAVE_API_KEY. Register alongside your custom servers. Researcher now has 3 tools: Brave Search (official MCP), Web Scraper (your MCP), Hybrid RAG Retriever. Log all tool calls with input/output/latency to agent_tasks.tool_calls JSONB column.
4.4
Build FastAPI layer in api/routes.py with async agent execution
POST /api/research — receives goal, creates Supabase research_run, starts LangGraph in background task, returns run_id immediately. GET /api/research/{run_id} — returns current status from Supabase. DELETE /api/users/{id}/data — GDPR right-to-erasure endpoint. This async pattern is what makes the live progress page possible.
4.5
Wire Streamlit Page 1 to FastAPI — replace direct agent call
Replace the direct agent call from Phase 1 with: response = httpx.post(f"{API_URL}/api/research", json={"goal": goal}), store run_id in st.session_state, redirect to Progress page. This decouples UI from agent execution and makes the live polling pattern work.
✓ Phase 4 Complete When: LangSmith shows your Researcher calling your custom scraper_mcp.py and citation_mcp.py — both servers you wrote. FastAPI accepts a research goal and returns a run_id within 200ms. The GDPR erasure endpoint returns 200 and deletes user data.

Week 5
06
Evaluation Suite — Get Your Benchmark Numbers
🧪 Evals
Goal: 40+ Q&A pairs evaluated. Before/after numbers documented. Numbers you can quote from memory in any interview.
6.1
Complete 20 manual Q&A pairs — write ground-truth answers from the actual documents
8 English EU AI Act questions, 6 English GDPR questions, 3 Swedish questions, 3 German questions. For each: read the source document, write the ground truth answer in your own words. Save as evals/qa_pairs_manual.json. Takes 1–2 hours. These are your golden dataset — the standard against which all RAGAS scores are measured.
6.2
Run RAGAS synthetic testset generator for 20+ additional pairs
from ragas.testset import TestsetGenerator; generator = TestsetGenerator.with_openai(); testset = generator.generate_with_langchain_docs(docs, test_size=20, distributions={simple:0.5, reasoning:0.3, multi_context:0.2}). Save as evals/qa_pairs_synthetic.json. Combined: 40+ pairs total. These auto-generated multi-hop questions test whether your retriever handles complex, cross-chunk queries.
6.3
Run RAGAS comparison: fixed-size chunking vs SemanticChunker
Temporarily swap to RecursiveCharacterTextSplitter(chunk_size=500). Re-embed the EU AI Act. Run RAGAS on all 40 pairs. Record faithfulness score. Then switch back to SemanticChunker. Re-run. Compare. The improvement proves your architecture decision was correct. This before/after is the core of your blog post.
6.4
Run LLM-as-Judge evaluation on 10 final reports
Run your demo goal 10 times (slight variations each time). Use GPT-4o to score each final report on: factual accuracy, completeness, citation quality, EU AI Act alignment (0–10 each). Store all scores in Supabase eval_scores. Calculate averages. These LLM-as-Judge scores go in your README alongside RAGAS scores.
6.5
Record your final benchmark numbers — memorise them
Write evals/baseline_results.json with: faithfulness score, answer_relevancy, context_precision, avg LLM-as-Judge score, avg cost per run in USD, avg latency in seconds, chunking improvement delta. Memorise the top 3 numbers. You will be asked "what were your evaluation results?" in every technical interview. Know the answer without opening your laptop.
✓ Phase 6 Complete When: You can say without hesitation: "My system achieved X faithfulness score on RAGAS with 40 Q&A pairs. SemanticChunker improved it by Y% over fixed-size chunking. Average cost per research run is $Z." Those numbers are your most valuable interview asset.
