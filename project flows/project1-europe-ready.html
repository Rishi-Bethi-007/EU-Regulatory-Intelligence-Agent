02 —— The 4 Upgrades — Step by Step
🔒
Upgrade 1 — GDPR Compliance Layer
Required by Law in EU
G.1
Set your Supabase region to eu-central-1 (Frankfurt)
In your Supabase project settings, select Frankfurt or Stockholm as the region. This ensures all user data stays within the EU. Add a comment in your README: Data residency: EU (Frankfurt, Germany). This one setting is legally significant and shows you know GDPR geography.
G.2
Create a consent_records table and data_subjects table in Supabase
Schema: consent_records(id, user_id, purpose, granted_at, ip_hash, version) and data_subjects(id, user_id, erasure_requested_at, anonymized_at). On Streamlit login, insert a consent record. This is minimal effort for maximum compliance signal.
G.3
Add a DELETE /api/users/{id}/data FastAPI endpoint — Right to Erasure
When called: anonymize the user's research_runs (null out user_id, hash the query text), delete their document_chunks, delete their uploaded files from Supabase Storage, insert a record into data_subjects. Log the erasure in your audit trail. This is GDPR Article 17 — right to be forgotten.
G.4
Add data minimization to the ingestion pipeline
In rag/ingest.py, before chunking: strip all personal data patterns (email regex, phone regex, EU national ID patterns) using a simple PII scrubber. Add pii_scrubbed: true to each chunk's metadata. This shows you think about data quality, not just model quality.
🇪🇺 Why This Gets You Hired Every European company processing user data needs GDPR compliance. Engineers who can implement it in their own projects are far more credible than those who only know it theoretically. This is a 2–3 hour addition with outsized interview impact.
🇪🇺
Upgrade 2 — EU AI Act Compliance Module
In Force Since Aug 2024
A.1
Add a risk classification function in compliance/risk_classifier.py
Based on the research topic, classify the AI use case as: UNACCEPTABLE / HIGH_RISK / LIMITED_RISK / MINIMAL_RISK per EU AI Act Annex III. Use a short prompt to Claude with the classification schema. Store the result in research_runs.risk_level. This is a 50-line module that shows deep EU AI Act awareness.
A.2
Build a transparency disclosure generator
For every research report, append a Transparency Notice section: which LLMs were used, what training data sources were consulted, confidence level, known limitations. Store as research_runs.transparency_notice. EU AI Act Article 13 mandates this for high-risk systems. Even doing it for limited-risk systems signals you read the regulation.
A.3
Add a 6th Streamlit page — EU AI Act Compliance Dashboard
Show: risk level badge per run, transparency score (0–100 based on how much metadata was captured), a checklist of EU AI Act obligations ticked off. Use st.metric for the transparency score. Add a one-click "Generate Compliance Summary" button that produces a PDF. This page is your portfolio centrepiece — screenshot it for LinkedIn.
A.4
Reframe your demo use case around EU AI Act research
Change your default demo goal from a generic topic to: "What are the EU AI Act obligations for a Swedish manufacturing SME deploying a quality control AI system?" — This is a real question thousands of companies are asking right now. Upgrade 3 — Explainable AI (XAI) Decision Traces
EU AI Act Art. 13
X.1
Add a DecisionTrace Pydantic model to every agent
Each agent node, alongside its output, must also return a DecisionTrace object with fields: agent_name, reasoning_steps: list[str], sources_used: list[str], confidence: float, alternatives_considered: list[str]. Store as JSONB in agent_tasks.decision_trace. This adds ~10 lines per agent and makes every decision auditable.
X.2
Show decision traces in the Report Viewer page
For each section of the final report, add an expandable st.expander("Why did the agent conclude this?") that shows the Analyst agent's reasoning_steps and sources_used. Add a confidence bar using st.progress(confidence_score). This is XAI made tangible — something most senior engineers talk about but few actually implement.
X.3
Add a Counterfactual Analysis tool to the Critic Agent
When the Critic scores a claim below 0.7, it now also generates: "If this claim is wrong, what would the alternative conclusion be?" — stored as counterfactual in the agent task. This is a genuine XAI technique (counterfactual explanations) and a strong interview talking point about model robustness.
🔍 The XAI Differentiator Most multi-agent systems are black boxes. Yours will have a full decision trace for every conclusion. In industries like healthcare, legal, and financial services — all huge in Europe — this is non-negotiable. Put "Explainable AI" prominently in your GitHub README and LinkedIn headline.
🌍
Upgrade 4 — Multilingual NLP (Swedish + German)
Local Language Edge
M.1
Replace OpenAIEmbeddings with multilingual-e5-large from HuggingFace
Install: pip install sentence-transformers. Use HuggingFaceEmbeddings(model_name="intfloat/multilingual-e5-large") from langchain_community.embeddings. This model supports 100+ languages including Swedish and German with state-of-the-art quality. It runs locally — no API costs, no data leaving your machine.
M.2
Add language detection in the ingestion pipeline
Install: pip install langdetect. After loading each document, run detect(doc.page_content[:500]). Store the result as metadata["language"] on each chunk. Add a language column to document_chunks. Add a language filter to your retriever — when the user's query is in Swedish, prioritise Swedish chunks.
M.3
Add a language selector to the Streamlit upload page
Simple st.selectbox("Document language", ["Auto-detect", "English", "Swedish (Svenska)", "German (Deutsch)", "Dutch (Nederlands)"]). Pass the selection to the ingestor. Show the detected language after upload as st.success(f"Detected language: Swedish 🇸🇪"). This is a 5-line UI addition that shows multilingual awareness.
M.4
Test with real Swedish regulatory documents
Download Swedish government PDFs on AI regulation from government.se or digg.se (the Swedish Agency for Digital Government). Upload them and verify your retriever returns relevant chunks for both Swedish and English queries. Include this test in your README. Screenshotting "Swedish PDF → correct Swedish answer" is worth more than 100 lines of code in a Swedish job application.
🌍 The Local Language Moat The vast majority of AI engineers applying to European roles speak only English and have built only English-language systems. The moment you show you've handled Swedish and German documents — even in a basic way — you've separated yourself from 80% of the competition. This matters especially in Stockholm, Munich, and Amsterdam.
