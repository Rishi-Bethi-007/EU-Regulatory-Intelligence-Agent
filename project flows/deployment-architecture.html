03 —— GitHub Actions CI/CD Pipeline — 4 Jobs
Trigger: push to any branch
Job 1: Test + Lint
runs-on: ubuntu-latest
Goal: Catch failures before any image is built
1
Checkout code + set up Python 3.11
2
Install dependencies from requirements.txt
pip install -r requirements.txt
3
Lint with Ruff (faster than flake8)
ruff check . --select E,W,F
4
Type check with mypy
mypy agents/ rag/ db/ api/ --ignore-missing-imports
5
Run unit tests with pytest
pytest tests/ -v --tb=short
Trigger: push to main branch only
Job 2: Build + Push to ECR
needs: [test-and-lint]
Goal: Build Docker image, tag with git SHA, push to AWS ECR
1
Configure AWS credentials from GitHub Secrets
aws-actions/configure-aws-credentials@v4
2
Login to Amazon ECR
aws-actions/amazon-ecr-login@v2
3
Build Docker image tagged with git SHA + latest
docker build -t $ECR_REPO:${{ github.sha }} -t $ECR_REPO:latest .
4
Push both tags to ECR
docker push $ECR_REPO:${{ github.sha }} && docker push $ECR_REPO:latest
5
Output image URI as job artifact for next job
Trigger: after ECR push succeeds
Job 3: Blue-Green Deploy
needs: [build-and-push]
Goal: Deploy to Green, health check, shift traffic, terminate Blue
1
Launch new EC2 instance (Green) from latest AMI
aws ec2 run-instances --user-data deploy_script.sh
2
Green instance pulls image from ECR + starts container
docker pull $ECR_REPO:${{ github.sha }} && docker run -d ...
3
Wait for ALB health check to pass on Green (max 5 min)
aws elbv2 wait target-in-service --target-group-arn $GREEN_TG
4
Shift 100% of ALB traffic to Green target group
aws elbv2 modify-listener --default-actions green_tg
5
Run smoke tests against live Green endpoint
curl -f https://api.yourdomain.com/health
6
On success: terminate Blue instance. On failure: rollback to Blue immediately.
Trigger: after deploy (always runs)
Job 4: Notify + Tag
needs: [deploy] · if: always()
Goal: Visibility into every deployment — success or failure
1
Post deploy status to Slack (or email) with git SHA + diff link
2
Create GitHub Release tag on successful deploy
gh release create v${{ github.sha }} --generate-notes
3
Log deployment event to Supabase deployments table

05 —— 4 Deployment Strategies — Know All of Them
Rolling Deployment
simplest
Replace instances one at a time. Old version serves traffic while new instances come up. If the new version fails, some users already got it.
Use when: downtime is okay and rollback speed doesn't matter. Not recommended for your project.
Blue-Green Deployment ★
recommended
Two identical environments — Blue (live) and Green (new). Deploy to Green, health check, switch ALB traffic instantly. Blue stays alive for instant rollback. Zero downtime.
Use for: your project. Clean cutover, instant rollback, zero downtime. Best resume talking point.
Canary Deployment
advanced
Send 5% of traffic to new version, monitor errors, gradually increase to 100%. Catches issues before they affect all users. Requires traffic splitting at ALB level.
Know for interviews. Overkill to implement for a portfolio project — mention it as "next step".
Recreate Deployment
avoid
Stop all old instances, start all new ones. Causes downtime between stop and start. Simple but unacceptable for any user-facing service.
Avoid. Mention it only to explain why you chose Blue-Green instead.

07 —— The Recommended Setup for Your Project

Your Deployment Stack — Final Answer

🚀
Backend → AWS EC2 + ALB
t3.medium, eu-north-1 (Stockholm). Single instance to start — ASG ready for when you need it. ~$35/mo. Worth every cent for the resume story.
🎈
Frontend → Streamlit Community Cloud
Free, zero config, auto-deploys from GitHub. Points to your EC2 backend URL. Don't overcomplicate the frontend deployment.
⚡
DB → Supabase (stay there)
Don't migrate to RDS. Supabase already handles pgvector, Auth, Storage, and Realtime. Moving to RDS costs weeks and adds nothing for a portfolio project.
🔄
CI/CD → GitHub Actions (full pipeline)
4 jobs: Test → Build+ECR → Blue-Green Deploy → Notify. Push to main = automatic deploy. Every merge is a deployment event with a paper trail.
🚢
Strategy → Blue-Green
Zero downtime, instant rollback. The single most impressive deployment talking point for a junior-to-mid AI Engineer role. Know it inside out.
🔐
Secrets → AWS Secrets Manager
All API keys (Anthropic, OpenAI, Supabase, Brave) stored in Secrets Manager. EC2 pulls them at startup via IAM role. Never in .env on the server.
